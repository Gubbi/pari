## ADDED Requirements

### Requirement: ChangeSet represents a substrate-agnostic set of entity changes
The system SHALL define a `ChangeSet` type in `src/substrate/changeset.rs` containing a flat `Vec<EntityChange>`. Each `EntityChange` SHALL carry:
- `path: String` — tree location (e.g., `"workflows/Initiative/WriteProposal"`)
- `kind: EntityKind` — the entity kind (Role, Hook, Team, Workflow, Task, Relay, SharedWorkflow)
- `id: String` — the entity's id
- `op: ChangeOp` — one of `Added(EntityData)`, `Modified { entity: EntityData, dirty_fields: Vec<String> }`, or `Removed`

`EntityData` SHALL be an enum carrying the full **tracked** entity value for each entity type (e.g., `EntityData::Role(TrackedRole)`). No conversion back to plain types is required — substrates receive tracked entities and access fields via `Deref`.

#### Scenario: Added entity in changeset
- **WHEN** a new Role `"eng-lead"` is added to the store and `collect_changes()` is called
- **THEN** the ChangeSet contains an `EntityChange` with `path: "roles"`, `kind: Role`, `id: "eng-lead"`, `op: Added(EntityData::Role(tracked_role))`

#### Scenario: Modified entity carries dirty field names and full tracked entity
- **WHEN** the `name` field of Role `"eng-lead"` is changed and `collect_changes()` is called
- **THEN** the ChangeSet contains an `EntityChange` with `op: Modified { entity: EntityData::Role(tracked_role), dirty_fields: ["name"] }`

#### Scenario: Removed entity carries full tracked value
- **WHEN** Role `"eng-lead"` is removed from the store and `collect_changes()` is called
- **THEN** the ChangeSet contains an `EntityChange` with `path: "roles"`, `kind: Role`, `id: "eng-lead"`, `op: Removed`

#### Scenario: Nested workflow step changes produce flat entries
- **WHEN** a Task `"WriteProposal"` nested inside Workflow `"Initiative"` has its `instructions` field modified
- **THEN** the ChangeSet contains an `EntityChange` with `path: "workflows/Initiative/WriteProposal"`, `kind: Task`, `id: "WriteProposal"`, `op: Modified { dirty_fields: ["instructions"] }`

---

### Requirement: EntityStore::collect_changes produces a ChangeSet without mutating state
The system SHALL provide `EntityStore::collect_changes(&self) -> ChangeSet` that walks all tracked entity collections and produces a flat `ChangeSet`. `collect_changes()` SHALL NOT reset dirty flags or modify any tracking state — it is a pure read operation. Calling `collect_changes()` twice with no intervening mutations SHALL return equivalent ChangeSets.

`dirty_fields` in `ChangeOp::Modified` SHALL be populated by calling `dirty_fields()` on the tracked entity — a method generated by `#[derive(Tracked)]` that walks each `Tracked<T>` field and returns the names of dirty ones.

#### Scenario: collect_changes collects all entity types
- **WHEN** changes exist across roles, hooks, and workflows
- **THEN** `collect_changes()` returns a ChangeSet containing entries for all three

#### Scenario: collect_changes does not reset dirty state
- **WHEN** `collect_changes()` is called
- **THEN** a subsequent `collect_changes()` with no mutations returns an equivalent ChangeSet

#### Scenario: collect_changes walks nested workflow steps
- **WHEN** a Workflow has a dirty nested Task three levels deep
- **THEN** `collect_changes()` produces a flat EntityChange with the full path to that Task

#### Scenario: Modified entry has accurate dirty_fields
- **WHEN** only the `name` field of a Role is mutated
- **THEN** `collect_changes()` produces `Modified { dirty_fields: ["name"] }` — not empty, not containing other field names

---

### Requirement: EntityStore::reset_tracked clears all dirty state after successful persistence
The system SHALL provide `EntityStore::reset_tracked(&mut self)` that clears all `inserted`, `modified`, and `removed` sets across all TrackedMaps (including nested workflow step maps), and resets all per-field dirty flags. `reset_tracked()` SHALL only be called after `atomic_persist()` confirms success.

#### Scenario: reset_tracked clears dirty state
- **WHEN** `reset_tracked()` is called after successful persist
- **THEN** `collect_changes()` returns an empty ChangeSet

#### Scenario: Failed persist leaves dirty state intact
- **WHEN** `atomic_persist()` returns an error and `reset_tracked()` is NOT called
- **THEN** `collect_changes()` still returns the same ChangeSet as before the failed persist attempt

---

### Requirement: RepoSubstrate uses LCA-based atomic persistence
The system SHALL compute the lowest common ancestor (LCA) directory of all changed file paths in the ChangeSet. Persistence SHALL stage changes within only the LCA subtree:
1. Create a `.part/` sibling directory of the LCA directory
2. Hard-link unchanged files within the LCA subtree into `.part/`
3. Write changed files (re-rendered) into `.part/`
4. Omit removed files from `.part/`
5. Rename the LCA directory to `.old/`, rename `.part/` to the LCA directory, delete `.old/`

If hard-linking fails with a cross-device error, the system SHALL fall back to file copy.

#### Scenario: Single entity change swaps only parent directory
- **WHEN** only `roles/eng-lead.md` changed and `roles/` contains 3 files total
- **THEN** only the `roles/` directory is swapped (2 hard-links + 1 write + rename)

#### Scenario: Changes in same subtree swap the common ancestor
- **WHEN** `workflows/Initiative/README.md` and `workflows/Initiative/WriteProposal/README.md` both changed
- **THEN** `workflows/Initiative/` is the LCA and only that subtree is swapped

#### Scenario: Changes across top-level directories swap root
- **WHEN** both `roles/eng-lead.md` and `workflows/Initiative/README.md` changed
- **THEN** root is the LCA and the entire root directory is swapped (equivalent to full snapshot)

#### Scenario: Initial persist with no existing root
- **WHEN** `atomic_persist()` is called and no root directory exists
- **THEN** all files are written to `.part/` and renamed to root (identical to current full-snapshot behavior)

#### Scenario: Empty changeset is a no-op
- **WHEN** `atomic_persist()` is called with an empty ChangeSet
- **THEN** no filesystem operations are performed

#### Scenario: Cross-device hard-link falls back to copy
- **WHEN** hard-linking a file fails with `EXDEV` (cross-device)
- **THEN** the file is copied instead and persistence continues normally
