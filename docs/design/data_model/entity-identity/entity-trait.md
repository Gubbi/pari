# entity-trait

**Data Model → `data_model/entity-identity/`**

---

## Purpose

`Entity` is the trait every entity type implements. It provides the runtime kind
discriminant, parent type, tracked counterpart, and validation schema — all as
associated items. Everything downstream (store, substrate, validators) reaches
these through `Entity`'s associated items rather than per-entity naming.

---

## `Entity` Trait

```rust
pub trait Entity {
    const KIND:              EntityKind;
    const VALIDATION_SCHEMA: ValidationSchema;
    type Parent:  ParentKind;
    type Tracked: TrackedEntity<Entity = Self>;
}
```

- `KIND` — runtime discriminant; used in `EntityRef` hash/eq, `AnyEntityRef` dispatch, substrate `load_strategy`
- `VALIDATION_SCHEMA` — static schema generated from `#[validations]` field attributes; see [codegen/validatable](../../codegen/validatable.md)
- `Parent` — `NoParent` for top-level entities; `WorkflowParent` for embedded entities whose identity includes a workflow parent chain
- `Tracked` — the store's mutable load-aware representation; `Role::Tracked = TrackedRole`

---

## `TrackedEntity` Companion Trait

```rust
pub trait TrackedEntity {
    type Entity: Entity<Tracked = Self>;
}
```

Implemented by every tracked entity. Provides the roundtrip:
`TrackedRole::Entity = Role`, and therefore `Role::Tracked::Entity = Role`.

---

## Type Alias

```rust
pub type Tracked<T: Entity> = <T as Entity>::Tracked;
```

`Tracked<Role>` is the ergonomic form used throughout the codebase in place of
`<Role as Entity>::Tracked`.

---

## Implementations

All impls are generated by `#[derive(Entity)]` — not hand-written:

```rust
// Top-level entity:
impl Entity for Role {
    const KIND:              EntityKind       = EntityKind::Role;
    const VALIDATION_SCHEMA: ValidationSchema = ROLE_VALIDATION_SCHEMA;
    type Parent  = NoParent;
    type Tracked = TrackedRole;
}

// Embedded entity — WorkflowParent instead of NoParent:
impl Entity for Task {
    const KIND:              EntityKind       = EntityKind::Task;
    const VALIDATION_SCHEMA: ValidationSchema = TASK_VALIDATION_SCHEMA;
    type Parent  = WorkflowParent;
    type Tracked = TrackedTask;
}
```

`TrackedRole` implements `TrackedEntity`:

```rust
impl TrackedEntity for TrackedRole {
    type Entity = Role;
}
```

The `entity_ref` field holds the concrete `EntityRef<T, P>` value. The derive
macro generates `Resolvable::to_any_ref()` for store operations that need
type-erased access (see [store-structure](../../store_layer/entity-store/store-structure.md)).

---

## Bound on `EntityRef`

```rust
pub struct EntityRef<T: Entity, P: ParentKind = NoParent> { ... }
```

`T: Entity` ensures only declared entity types can be referenced.
`EntityRef<Raci>` or `EntityRef<String>` is a compile error.

`P = NoParent` default covers top-level entities — the parent is written
explicitly only for hierarchy-bearing entities: `EntityRef<Task, WorkflowParent>`.

---

## Plain and Tracked Entities

Only plain entity structs implement `Entity`. Tracked entities implement
`TrackedEntity`. `TrackedRole` is reachable as `Role::Tracked`; `Role` is
reachable from a tracked entity as `Tracked<Role>::Entity`.

Plain entities carry structurally valid data by construction — required fields
present, types correct. Runtime validation (semantic, cross-entity) runs against
tracked entities only, where field access is async and load-aware.

`entity_ref` is a concrete field on both plain and tracked entities — not a
trait method. Type-erased access uses `AnyEntityRef`
(see [any-entity-ref](any-entity-ref.md)).

The `Parent` associated type defines relationship semantics, while the Store
still treats every entity as a standalone record keyed by its full `EntityRef`.
